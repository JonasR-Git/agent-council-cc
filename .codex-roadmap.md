## Phase C — Robustness & token economy

C1. **Schemas for all structured outputs.** Add to plugins/council/schemas/:
    critique-votes.schema.json (R2 peer critique: agent/about/summary/votes[targetId,title,vote,note]/missed[]),
    plan-critique.schema.json (agent/about/summary/scores{feasibility,risk,simplicity,completeness}/overall/blockers[]/improvements[]),
    debate-rebuttal.schema.json (id/stance/note/revisedSeverity), debate-counter.schema.json (id/upheld/note).
    Keep them draft-2020-12 like the existing two.
C2. **Tiny recursive validator.** plugins/council/scripts/lib/validate.mjs (zero-dep, target <=150 lines):
    supports exactly: type (incl. ["string","null"] unions), required, properties (recursive),
    items (recursive), enum, minimum, maximum. Export validate(schema, value) ->
    { valid, errors: ["path: message"] }. No $ref support needed (inline the schemas accordingly).
C3. **Wire validation with per-type semantics** (verified critique requirement — do NOT use one
    blanket rule):
    - findings docs (R1/claude file): schema-invalid -> parseOk:false + validationErrors[] on the
      doc; report shows "Structured parse failed" plus the first 3 error strings.
    - plan docs: same semantics as findings (excluded from ranking; errors in report).
    - R2 critique votes: invalid vote entries are dropped individually; doc-level invalid -> ignored
      with a note in the report ("critique ignored: <first error>"). Valid votes still apply.
    - plan critiques: invalid -> excluded from scoring with a report note.
    - debate rebuttal/counter: invalid -> treated as stance "defend" / upheld true (current
      fallbacks) but annotated parseOk:false.
    Loading schemas: read the JSON files once (fs) from the schemas dir.
C4. **Compact results.** council-companion `result` gains `--summary` flag:
    - deliberate jobs: print ONLY the "## Merged findings" section + snapshot header lines
      (target/branch/critique-scope) + debate section if present.
    - solve jobs: print ONLY problem heading, "## Ranking", blockers lists, and the synthesis
      instructions section.
    - review jobs: per-agent status lines (backend/model/exit) + first 40 lines of each agent body.
    - pending/failed: unchanged behavior.
    Implement by storing structured sections at job-finish time: job.reportSections =
    { header, merged, debate, ranking, full } (strings; null where n/a) — do NOT regex-parse the
    report at read time. `--summary --json` returns { jobId, status, sections } without raw stdout.
    Update plugins/council/commands/result.md to document --summary.
C5. **Sidecar artifacts + slim job JSON.** At job-finish, write each agent's raw stdout/stderr to
    `<stateDir>/artifacts/<jobId>/<agent>-<round>.txt` (round: r1|r2-<about>|debate-<id>|review).
    In job JSON results[], truncate stdout/stderr to 4000 chars each and add artifactFile paths.
    The full report string stays in the job (it is the primary artifact). Add stateVersion: 2 to
    job JSON; readers must not break on old jobs (absence of fields is fine). pruneJobs also removes
    the job's artifacts dir. Tests: artifacts written, truncation applied, prune removes dir,
    old-format job files still list fine.
C6. **Fence-aware truncation for sidecar-truncated stdout in reports** is NOT needed (reports embed
    the full stdout as before at render time, BEFORE truncation) — make sure report generation uses
    the in-memory full stdout, not the truncated copies.
